From vision to version number

A methodology for decomposing software projects into architecture, design, tasks, and releases — with a worked example.

This document walks through each decomposition level using a single example project: a SaaS platform for freelancer invoicing — create invoices, send them, get paid, track revenue. Classic mid-complexity SaaS (Next.js, Postgres, Auth, Stripe, email, background jobs).

At each level we show just one representative slice. The goal is to demonstrate the method, not plan the whole project.

LEVEL 1

Vision

The vision is a long-term hypothesis for the project. It's not a feature list — it's a bet. Everything below it exists to test whether this bet is correct.

The vision doesn't map to any version number. It's above that layer entirely. It answers why this project exists and how you'll know it's working.

LEVEL 2

MVP & milestones

Pick the minimum set of milestones that test the hypothesis. Everything above the MVP line is v1.0. Everything below is future minor versions.

Key insight: Milestones are capability checkpoints. They answer "what can the user do now that they couldn't before?" Each milestone is a meaningful unit of value, not just a bag of tasks.
LEVEL 3

Architecture (per milestone)

LEVEL 4

Detailed design (per component)

Zoom into one component from the architecture — the Stripe adapter — and define its interfaces, data model, happy path, and error states. This is the last design artifact before code.

One component, fully specified. Every other component in the architecture gets the same treatment. The detailed design answers "if I handed this to a developer, could they build it without further questions?"

LEVEL 5

Tasks, phases & version mapping

This is where the planning hierarchy meets the delivery hierarchy. Tasks come from the detailed design. Phases are how you order them. Versions are what you stamp on releases.

The mapping rules

majorYou broke the public contract (API, schema, auth). Usually an architectural epoch shift.
minorNew capability shipped. Often aligns with a milestone completion.
patchFix within a milestone. Maps to one or a few tasks.
date-commitBuild metadata. Maps to nothing in planning — it's CI/CD traceability.
Phases are invisible to users. Phase A/B/C within M2 are how you organise work. They never appear in a version number. A user doesn't care that you built the DB schema before the webhook handler — they care that "payments work" (minor bump).

The full picture

The flow is always top-down for planning, bottom-up for building. You decompose Vision → Milestones → Architecture → Design → Tasks. But you build tasks → phases → milestones → releases. The version number is stamped at the release boundary, which is a delivery decision, not a planning one.

Keep these as loosely coupled systems with clear mapping points. Milestones define "what's done." Releases define "what ships and with what compatibility promises." Sometimes they coincide perfectly. Often they don't — you might ship two releases within one milestone, or a single release might span parts of two milestones.

Plan in milestones and phases. Release when the code is ready and stable. Version according to semver's actual rules. Use the metadata suffix for build traceability. The planning board and the version number both tell true stories — they're just narrating different things.
